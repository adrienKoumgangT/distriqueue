# rust-worker

Rust-based queue worker for DistriQueue.
It consumes prioritized RabbitMQ jobs, executes supported job handlers, and sends status/heartbeat updates over HTTP.

## What it does

- Connects to RabbitMQ and consumes from 3 queues:
  - `job.high`
  - `job.medium`
  - `job.low`
- Enforces strict priority order (`high` -> `medium` -> `low`) using a biased `tokio::select!` loop.
- Limits concurrent processing with a semaphore (`WORKER_CAPACITY`).
- Sends job status updates to the API endpoint in `STATUS_UPDATE_URL`.
- Sends periodic worker heartbeat updates every 10 seconds.

## Tech stack

- Rust (edition 2024)
- Tokio (async runtime)
- Lapin (RabbitMQ client)
- Reqwest (HTTP client)
- Serde / Serde JSON
- Env Logger / Log

## Configuration

Environment variables (see `.env`):

- `RABBITMQ_HOST` (default: `10.2.1.11`)
- `RABBITMQ_PORT` (default: `5672`)
- `RABBITMQ_USERNAME` (default: `admin`)
- `RABBITMQ_PASSWORD` (default: `admin`)
- `WORKER_CAPACITY` (default: `10`)
- `WORKER_ID` (optional; autogenerated if not set)
- `STATUS_UPDATE_URL` (default: `http://10.2.1.11:8080/api/jobs/status`)
- `ORCHESTRATOR_URL` (defined in `.env`; see note in Known caveats)

## Run locally

1. Make sure RabbitMQ and your status/orchestrator services are reachable.
2. Set environment variables (for example by loading `.env`).
3. Start the worker:

```bash
cargo run
```

For logs:

```bash
RUST_LOG=info cargo run
```

## Queues and exchange

At startup, the worker declares and binds:

- Queue `job.high` with max priority `10`
- Queue `job.medium` with max priority `5`
- Queue `job.low` with max priority `1`

Each queue is bound to exchange `jobs.exchange` with its queue name as routing key.

## Job format

Expected JSON payload:

```json
{
  "id": "job-123",
  "type": "calculate",
  "payload": {  },
  "retry_count": 0,
  "max_retries": 3
}
```

Fields:

- `id` (string, required)
- `type` (string, optional, defaults to `default`)
- `payload` (object, optional)
- `retry_count` (number, optional)
- `max_retries` (number, optional)

## Supported job handlers

### 1) Calculation handler

Handled types: `calculate`, `sum`, `average`

Payload format:

```json
{
  "operation": "sum",
  "numbers": [1, 2, 3]
}
```

Supported operations:

- `sum`
- `average`

### 2) Transformation handler

Handled types: `transform`, `uppercase`, `lowercase`, `reverse`, `filter`, `normalize`

Payload format:

```json
{
  "operation": "uppercase",
  "data": {
    "name": "alice",
    "city": "paris"
  }
}
```

Supported operations:

- `uppercase`
- `lowercase`
- `reverse`
- `filter` (`filter_key` / `filter_value` optional)
- `normalize` (normalizes numeric values in `data`)

## Status + heartbeat payloads

- Job status updates include: `jobId`, `status`, `workerId`, `timestamp`, and optional `result` / `errorMessage`.
- Heartbeats include: worker identity/type, capacity/load metrics, processed counters, and timestamp.
